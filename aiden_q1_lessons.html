Sprites, Stage & Events
The rover gets eyes and a body — Aiden meets Scratch for the first time
EVENTS SEQUENCING SPRITES
One New Concept
Events trigger actions
Mission Theme
Rover boot sequence
Success Signal
Sprite moves on key press

"Welcome to Training Academy, Rover Pilot. Before we launch to Mars, we have to boot up the rover's operating system. Today you'll teach the rover to listen to commands — when you press a button, the rover responds. That's called an event. Events are how every computer program begins."

⏱ Session Timeline
0:00–0:05
Mission Briefing
Show Aiden a 60-sec NASA rover video. "What do you think tells it to move?"
0:05–0:15
Introduce Scratch
Open scratch.mit.edu. Tour: Stage, Sprites panel, Block palette. Name each part together.
0:15–0:55
Build: Rover Boot Sequence
Choose a rover sprite (rocket or robot). Add "when green flag clicked → say Hello, I'm your rover!" then "move 10 steps." Extend with sounds and looks.
0:55–1:05
Debugging Round
Change one block. Ask: "What do you predict happens now?" Run it. Compare.
1:05–1:10
Debrief
Aiden explains the project to you. "What did the rover learn today?"
🧠 Concepts to Teach
▸
Sprite — the character/rover that does things
▸
Stage — the world the rover lives in (Mars background)
▸
Event — something that causes a reaction ("when flag clicked")
▸
Sequence — blocks run top to bottom, in order
▸
Block categories — Motion (blue), Events (orange), Looks (purple)
Today's Scratch Blocks to Explore
when 🚩 clicked
move (10) steps
say [Hello!] for (2) secs
play sound (meow)

Before building

"That's called an event! The green flag is a signal, and the rover heard it. You programmed a response!"
When something breaks
"Interesting — the rover didn't do what we expected. What block do you think controls the movement? Let's find it."

Order of blocks matters
Mission Theme
Navigate obstacle field

⏱ Session Timeline
0:00–0:05

Ask Aiden to open his Week 1 project and explain it to you in 60 seconds.
0:05–0:15
Concept: Sequences

0:15–0:55
Build: Mars Navigation
Set Mars backdrop. Draw boulders with Scratch paint tool. Program exact move/turn sequence to reach a star sprite (the research station). Use degrees for turning.

1:05–1:10
Debrief
"Why did the order of the blocks matter today?"
🧠 Key Lesson: Order Matters
The core insight of week 2: swapping blocks changes behavior. Prove it.
Version A — Wrong Order
turn ↻ (90) degrees move (100) steps
→ Rover drives into boulder!
Version B — Right Order
move (100) steps turn ↻ (90) degrees
→ Rover clears boulder, then turns ✓
Ask: "What was different? We used the same blocks — just in a different order."
💬 Parent Script
Before coding (paper phase)
"Before we touch Scratch — can you draw the path on paper first? What are the exact steps the rover needs?"
When rover goes wrong way
"Interesting data! The rover went right instead of forward. Which block do you think caused that?"
End of session
"If I were a robot with no brain — exactly what would you tell me to do to walk from this chair to the door?"

Week 3 · Computational Foundations
Loops — Repeat Forever
The rover learns to patrol — doing the same thing again and again without being told each time
LOOPS REPEAT FOREVER
One New Concept
Loops = repeat automatically
Mission Theme
Rover patrol route
Success Signal
Rover loops patrol without extra blocks

"Mission Control needs the rover to patrol the research perimeter — over and over — watching for dust storms. But we can't keep sending it commands. We need the rover to repeat the patrol on its own. That's a loop: a superpower that makes computers do repetitive work so we don't have to."

⏱ Session Timeline
0:00–0:05
Debrief Week 2
Aiden re-runs his navigation mission. "What would happen if we had to do the whole path 10 times?" Show the pain of no loops.
0:05–0:15
Concept: Loops (unplugged)
Physical activity: Tell Aiden to clap 5 times. Then: "Now clap until I say stop." Which instruction was easier to give? That's a loop.
0:15–0:50
Build: Patrol Route
Start without loops — write 20 blocks for 5 patrol laps. Then delete all and replace with "repeat(5)" or "forever." Feel the power.
0:50–1:05
Explore: repeat vs forever
What's different? repeat(5) stops. forever never stops. When would you use each? (Patrol vs. always watching for danger)
1:05–1:10
Debrief
"Explain loops to me as if I've never heard the word."
🧠 Concepts to Teach
▸
repeat (n) — do something exactly n times, then stop
▸
forever — do something endlessly until the program stops
▸
Loop body — the blocks inside the loop that get repeated
▸
Why loops matter — computers are good at repetition; programmers aren't
Without Loop — 20 blocks for 5 laps
move 100 steps turn 90° move 100 steps turn 90°
... ×5 more times 😩
With Loop — 3 blocks for 5 laps ✨
repeat (5)
move 100 steps turn 90°
💬 Parent Script
Introducing loops (unplugged)
"I want you to take one step forward. Now take another step. Now another… [keep going]. This is annoying, right? What if I could just say 'take 10 steps' once? That's a loop."
When building repeat block
"Before you run it — how many times do you predict the rover will go around? Let's count together when it runs."
forever vs repeat
"When would you want the rover to patrol forever? When would you want it to stop after 5 times? What's the difference in the mission?"
✅ Success Criteria
✓
Aiden built the patrol route first without loops (feels the pain)
✓
Then replaced it with a loop and can explain why it's better
✓
Predicts correctly how many times rover will loop before running
✓
Can explain the difference between "repeat(5)" and "forever"
🌟
Stretch: Nest a loop inside a loop — rover spirals outward (patrol expanding perimeters).

Week 4 · Computational Foundations
Loops — Loop Challenge
Consolidate loops by solving increasingly complex patrol patterns. Mastery through iteration.
LOOPS PATTERNS MASTERY
Focus
Loop mastery challenges
Mission Theme
Geometric patrol patterns
Success Signal
Draws a square with 4-line loop

"The rover needs to map the Mars terrain with precise geometric patterns — squares, triangles, hexagons. Each shape can be drawn with a loop. How few blocks can you use? Mission Control rewards efficiency."

🟣 Challenge 1
Draw a Square
Use exactly 1 loop. 4 sides. Each side = 100 steps, each turn = 90°.
repeat (4)
move 100 steps turn 90°
Ask first: "What number goes in repeat? How do you know?"
🟣 Challenge 2
Draw a Triangle
Same idea, different numbers. 3 sides. Turn angle = 120°. Why 120?
Teach: 360° ÷ number of sides = turn angle.

Triangle: 360 ÷ 3 = 120°
Square: 360 ÷ 4 = 90°
Let Aiden figure out hexagon (360÷6=60°) on his own.
🟣 Challenge 3
Draw a Circle (Spiral)
Many small moves + small turns = circle. The big reveal: loops can approximate curves.
repeat (36)
move 10 steps turn 10°
Mind-blowing moment: "It's not actually curved — it's 36 tiny straight lines!"
💬 Parent Script
Before challenge 2
"Before you touch Scratch — how many sides does a triangle have? So how many times should the loop repeat? And if a full circle is 360°, what's the turn for 3 sides?"
Discovery moment
"You just used math to program geometry. NASA engineers do exactly this to plan rover paths."
✅ Success Criteria
✓
Draws square using repeat(4) without help
✓
Calculates triangle turn angle using 360÷3 reasoning
✓
Attempts hexagon or pentagon independently
✓
Can explain loop body vs. loop count distinction verbally

Week 5 · Computational Foundations
Conditionals — If/Then
The rover gets a brain — it can now make decisions based on what it senses
CONDITIONALS IF/THEN SENSING
One New Concept
If condition → action
Mission Theme
Obstacle detection system
Success Signal
Rover stops when it hits edge

"The rover is driving autonomously — but what happens when it hits a rock? It needs to sense the obstacle and decide what to do. That's a conditional: IF something is true, THEN do something. Every smart machine in the world runs on conditionals."

⏱ Session Timeline
0:00–0:05
Real-life conditionals
Ask: "What do you do if it's raining? If you're hungry? If a traffic light is red?" Conditionals are everywhere.
0:05–0:15
Introduce if/then block
Show the yellow "if...then" block. Inside goes the condition (sensing). Outside goes the action. Draw it on paper first.
0:15–0:50
Build: Edge Detection
"if touching edge → turn 180° and say 'Obstacle detected!'" Wrap in forever loop. Rover now bounces around Mars on its own.
0:50–1:05
Extend: Add color sensing
Draw a red "danger zone" on Mars backdrop. "if touching color red → stop and flash SOS." Add second conditional.
1:05–1:10
Debrief
"What's the difference between the rover always turning vs. only turning when it touches something?"
🧠 Concepts to Teach
▸
Condition — something that's either TRUE or FALSE
▸
if...then — only runs the inner blocks when condition is true
▸
Sensing blocks — how the program detects the world (touching edge, touching color, key pressed)
▸
Loop + conditional — the forever loop keeps checking; conditional decides each time
Rover Obstacle Detection
when 🚩 clicked forever
move 3 steps if <touching edge?> then
turn 180° say [Obstacle!] 1 sec
💬 Parent Script
Building intuition first
"Before Scratch — I'll walk toward the wall. When should I turn? IF I'm about to hit the wall THEN I turn. That's what we're teaching the rover."
After it works
"The rover is making a decision on its own now. You didn't tell it to turn at 3 o'clock — you told it the rule. It figured out the rest."
✅ Success Criteria
✓
Rover detects and responds to edge collision autonomously
✓
Can explain: "The if block checks — if true, it does this; if false, it skips"
✓
Added a second conditional (color detection or key press) independently
✓
Explains difference between always doing vs. doing only when condition met

Week 6 · Computational Foundations
If/Else — Two-Path Decisions
The rover can now choose between two options — not just react, but decide
IF/ELSE BRANCHING DECISION TREES
One New Concept
Two branches, one choice
Mission Theme
Day/night mode switching
Success Signal
Rover behaves differently in 2 conditions

"Mars has day and night. During the day, the rover collects samples. At night, it goes into low-power mode and flashes a beacon. The rover needs to check: is it day OR night? And do something DIFFERENT for each. That's if/else — the fork in the road."

⏱ Session Timeline
0:00–0:05
Debrief + Real-life else
"If you're hungry, eat a snack. ELSE, keep playing." Model the full if/else structure verbally before Scratch.
0:05–0:15
if/else block intro
Show how the if/else block has two pockets. One for when condition is TRUE, one for FALSE. Compare to if/then which only has one.
0:15–0:50
Build: Day/Night Rover
Add a backdrop with day (light) and night (dark). Use a variable "isDay" (true/false) or a key-press toggle. Rover samples during day, beacons at night.
0:50–1:05
Extend: 3-way decision
Add a third condition — dust storm! if dustStorm → shelter. Nested if/else preview.
1:05–1:10
Debrief
"Draw me the decision tree the rover uses. Show me the forks."
🧠 The Decision Tree Visual
Draw this together on paper before coding:
      IS IT DAY?
    /         \
  YES       NO
   |          |
COLLECT  BEACON
SAMPLES  FLASH
The key insight: both paths happen — the rover always does SOMETHING. if/then could leave the rover doing nothing half the time.
💬 Parent Script
Drawing the decision tree
"Before we code — draw me what the rover should do. If day… draw an arrow. What does it do? Now if NOT day — draw the other arrow. What now?"
After building
"Your rover can make choices now. Before, it could only react. Now it can decide. What's the difference?"
✅ Success Criteria
✓
Drew a decision tree on paper before coding
✓
Rover does different things in two distinct states
✓
Can explain: "The else block runs when the condition is false"
✓
Distinguishes if/then from if/else unprompted

Week 7 · Computational Foundations
Variables — Rover Telemetry
The rover starts tracking data — fuel, distance, battery — numbers that change over time
VARIABLES DATA COUNTERS
One New Concept
Named boxes that store numbers
Mission Theme
Rover fuel & battery tracker
Success Signal
Fuel decreases, rover reacts

"Mission Control is blind — they don't know how much fuel or battery the rover has. We need to build a telemetry system: variables that track fuel level, distance traveled, and battery percentage. When fuel hits zero, the rover shuts down. Variables are labeled boxes that hold changing numbers."

⏱ Session Timeline
0:00–0:10
Physical analogy
Get a jar and label it "FUEL." Put 10 beans in it. Each move the rover makes, remove one bean. When empty = shutdown. This is a variable.
0:10–0:20
Create variables in Scratch
Variables → Make a variable → "fuel" (set to 100). Show on screen. Set, change, and read the variable. Make fuel show on Stage.
0:20–0:55
Build: Telemetry Dashboard
Rover moves in forever loop. Each loop: change fuel by -1. If fuel = 0 → stop all, say "Mission aborted!" Aiden watches the countdown.
0:55–1:05
Extend: Score counter
Add a "samples" variable. When rover touches a yellow sprite (sample), samples goes up by 1. Now rover has a goal!
1:05–1:10
Debrief
"What's the variable storing right now? What changes it? What reads it?"
🧠 Variables = Named Boxes
The mental model that works for 7-year-olds:
VARIABLE = LABELED BOX
📦 Box name: fuel
📦 Contents right now: 87
📦 Contents can change: YES, every loop
Telemetry Loop
forever
move 2 steps change [fuel] by (-1) if <fuel = 0> then
stop [all]
💬 Parent Script
Jar analogy setup
"This jar is the rover's fuel tank. Each time it moves, it uses fuel. We're going to build a Scratch version of this exact jar. The variable IS the jar."
When fuel hits 0
"What just happened? Why did it stop? Which block caused that? How would you make it last longer?"
✅ Success Criteria
✓
Created a variable and can explain what it stores
✓
Fuel variable decrements each loop and triggers shutdown at 0
✓
Can adjust the starting fuel value and predict how long rover runs
✓
Added a second variable (samples or battery) independently
🌟
Stretch: Add a "refuel" station sprite. When rover touches it, fuel resets to 100.
